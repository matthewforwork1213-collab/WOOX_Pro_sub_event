### 도메인 컨텍스트

**서비스 개요**: TetherMax 플랫폼 내 유저 행동 접점(출금 완료 화면, 캐시백 프리뷰 결과 페이지)에 WOOX Pro 전환 유도 넛지를 추가하는 프로모션 기능.

- **기능 1 — 출금 완료 후 WOOX Pro 프로모션 분기 노출**: WOOX Pro 미포함 출금 시, 기존 이벤트 광고 대신 "WOOX Pro 사용 시 얼마를 더 받을 수 있었는지" 가상 피드백 노출.
- **기능 2 — 캐시백 프리뷰 결과 내 WOOX Pro 비교 모달 강제 노출**: 페이백 미리보기에서 타 거래소 선택 시 배경 Dim 처리 + WOOX Pro 비교 모달 팝업 강제 노출.

### 도메인 용어

| 용어 | 설명 |
|---|---|
| 넛지 (Nudge) | 유저를 강제하지 않고 특정 행동을 유도하는 UI/UX 설계 기법 |
| 손실 회피 편향 | 동일한 금액의 이익보다 손실을 더 크게 인식하는 심리 편향 |
| 가상 피드백 | "WOOX Pro를 사용했더라면 얼마를 더 받을 수 있었는지" 계산된 반사실 문구 |
| 딤(Dim) 처리 | 배경 화면을 반투명하게 흐리게 만드는 UI 처리 |
| 페이백 미리보기 | 유저가 거래소 조건을 입력하면 예상 캐시백을 미리 계산해 보여주는 기능 |
| 프로모션 분기 | WOOX Pro 포함/미포함 여부에 따라 노출 콘텐츠를 다르게 처리하는 로직 |
| CTA 버튼 | Call To Action — 유저를 WOOX Pro 가입/상세 페이지로 이동시키는 버튼 |
| 세션 노출 제한 | 동일 세션 내 비교 모달을 1회만 노출하는 피로도 방어 로직 |
| 커미션율 | 파트너 계정(테더맥스)이 거래소로부터 받는 수수료 분배 비율 (WOOX Pro 90%, 내부 전용) |
| 페이백율 | 커미션 중 유저에게 캐시백으로 환급하는 비율 (WOOX Pro 80%, 내부 전용) |

### 산출물 HTML 출력 파일명 규칙 (덮어쓰기 방지)

재생성 가능한 HTML 산출물(화면설계서 등)의 파일명에는 타임스탬프(YYYYMMDD_HHMM)를 반드시 포함합니다.

| 산출물 | 파일명 예시 |
|---|---|
| 화면설계서 HTML | 화면설계서_20260630_1430.html |

- 사용자가 명시적으로 "덮어써줘"라고 요청하지 않는 한 타임스탬프로 신규 저장합니다.
- `.md` 원본은 단일 파일로 유지하며 타임스탬프 규칙을 적용하지 않습니다.

## 핵심 비즈니스 규칙 (산출물 작성 시 반드시 준수)

1. **역효과 방지**: 타 거래소 페이백이 WOOX Pro보다 크게 계산되는 경우 광고 및 모달 절대 미노출.
2. **기준 명시 의무**: 모든 수수료 비교는 '일반 유저/기본 등급' 기준임을 모달 하단에 소문자로 명시.
3. **세션 피로도 제한**: 비교 모달은 세션당 최대 1회 노출. 중복 노출 금지.
4. **WOOX Pro 선택 시 모달 미노출**: 유저가 이미 WOOX Pro를 선택한 경우 딤 처리 및 비교 모달 표시 불가.
5. **CTA 브랜드 컬러 준수**: Primary CTA 버튼은 TetherMax 브랜드 컬러 적용.
6. **내부 수치 비노출**: 커미션율(90%)·페이백율(80%) 원자료 및 예산 등 내부 수치는 유저 화면에 절대 표시 금지. 유저에게는 계산 결과값(절약 금액 USDT, %p)만 노출. (예산 등 대외 공개 카피는 비전 프레이밍 사용)

## Gate-Check Rules (V0.43)

산출물 완료 시 아래 5단계 프로세스를 반드시 준수합니다.

1. **사전 확인**: 산출물 작성 전 `.progress.md`를 읽고 전제조건(상태=완료) 확인
2. **미충족 시**: 사용자에게 미충족 항목을 안내하고, 선행 산출물 작성을 제안
3. **완료 처리** (산출물 작성 후 순서대로 실행):
   - (a) `.progress.md`의 상태를 "완료"로, 완료일을 기록
   - (b) 가능하다면 NotebookLM에 해당 산출물을 소스로 등록
   - (c) 등록 실패 시 `.progress.md` 비고란에 `[NLM 미등록]` 기록 후 다음 작업 계속
4. **실체 검증**: 테스트결과보고서(#16)는 `tests/` 파일 존재 확인, 완료보고서(#17)는 15개 이상 산출물 완료 확인
5. **강제 스킵**: 사용자가 "gate-check 무시하고"라고 명시할 경우 경고 후 진행, `.progress.md`에 `[SKIP]` 태그 기록

### Gate-Check 품질 기준

| 항목 | 기준 |
|---|---|
| 한국어 작성 | 모든 산출물은 한국어 (영문 용어는 괄호로 설명) |
| 이모지 미사용 | 이모지 금지 |
| 표 활용 | 복잡한 정보는 표로 구조화 |
| OI 태깅 | 미결 사항은 OI-## 형식으로 표시 |

## Output Format

### 마크다운 작성 기준

- **제목**: `#`, `##`, `###` 레벨만 사용 (4단계 이상 금지)
- **목록**: 순서 없음은 `-`, 순서 있음은 `1.` (혼용 금지)
- **강조**: 볼드 `**text**` 사용
- **코드**: 인라인은 `` `code` ``, 블록은 ` ```lang `
- **표**: 최소 2열 이상 권장

### 문서 헤더 템플릿

모든 산출물의 첫 섹션:

## Git Conventions

### 커밋 메시지 형식

```
[영역] 간단한 설명

상세 내용 (필요 시)
- 항목 1
- 항목 2

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

**영역 (Scope):**
- `docs:` — 산출물 문서 추가/수정
- `feat:` — 기능 개발 (코드)
- `fix:` — 버그 수정
- `chore:` — 설정, 폴더 구조
- `test:` — 테스트 코드

### 브랜치 전략

| 브랜치 | 용도 | 생성 규칙 |
|---|---|---|
| `main` | 프로덕션 | PR 병합으로만 업데이트 |
| `develop` | 개발 통합 브랜치 | PR 리뷰 후 병합 |
| `feature/*` | 기능 개발 | `develop`에서 생성 |
| `docs/*` | 산출물 작성 | `develop`에서 생성 |
| `fix/*` | 버그 수정 | `develop`에서 생성 |
