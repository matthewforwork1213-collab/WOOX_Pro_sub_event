## 2. 세부 기능 요구사항 (Functional Requirements)

### 기능 1: 출금 완료 후 WOOX Pro 프로모션 분기 노출

**기존 로직**

유저가 (여러 거래소 동시 출금을 포함하여) 출금을 완료하면, 화면 내 우선순위에 따라 현재 진행 중인 거래소 이벤트 중 임의의 1개를 광고 형태로 노출.
> 예: 토스 송금 후 보험 광고 형태

**변경 및 신규 로직 (30일 한정)**

**케이스 A: WOOX Pro가 포함되지 않은 출금 건**

- 기존의 일반 이벤트 광고 대신, **"만약 WOOX Pro를 사용했더라면 이번에 수수료를 얼마나 더 아낄 수 있었는지"**(토탈 세이빙 기준, OI-08 확정) 계산된 가상 피드백 문구를 광고 영역에 노출
- **UI/UX:** 시각적으로 유저가 이득을 놓쳤다는 느낌(기회비용)을 주는 카피라이팅 적용

**케이스 B: WOOX Pro가 포함된 출금 건**

| 우선순위 | 조건 | 노출 내용 |
|------|------|-----------|
| 1 | WOOX Pro 자체 진행 이벤트가 **있을** 시 | **WOOX Pro 이벤트**를 최우선 노출 |
| 2 | 자체 이벤트 없음 & 테더맥스 WOOX Pro 온보딩 이벤트 활성 | **테더맥스 WOOX Pro 온보딩 이벤트** 노출 |
| 3 | 1·2 모두 없을 시 | **테더맥스 일반 하우스 광고** 노출 |

(상세 분기 로직은 §2-A-3 참조)

> **같은 화면에 기능 3도 함께 노출된다**: 이 출금 완료 화면(S1)의 결과 카드 영역 바깥 하단에는 **기능 3(WOOX Pro × 빗썸 트레블룰 연동 배너, 위치 #2)**이 케이스 A/B 분기와 별개로 노출된다. 노출 여부는 기능 1과 동일한 공통 프로모션 활성 게이트(§3-3)를 따른다. 상세는 기능 3 섹션 참조.

### 기능 2: 캐시백 프리뷰(페이백 미리보기) 결과 페이지 내 WOOX Pro 비교 카드 인라인 노출

> **변경 이력**: 기존 "딤 처리 + 강제 팝업 모달" 방식에서 **결과 페이지 자체를 인라인 변형**하는 방식으로 전환(OI-04 결정 반영). 팝업/딤 제거로 강제성·이탈률을 낮추고, 결과 흐름 안에서 손실 대비를 자연스럽게 노출한다.

**진입 맥락**

유저가 '페이백 미리보기'에서 다음 조건을 입력한 후 **[결과 페이지]에 진입한 시점**에 작동:
- 거래소
- 자산 규모
- 레버리지
- 거래 타입
- 거래 빈도

**작동 로직**

**케이스 A: 선택한 거래소가 WOOX Pro가 아닌 경우 (예: Zoomex, BitMart 등)**

1. 팝업·딤 처리 없이, 기존 결과 카드(선택 거래소의 거래 수수료·페이백 요율·예상 페이백)는 그대로 유지
2. 그 **바로 아래에 [WOOX Pro 비교 카드]를 인라인으로 삽입**하여 "현재 vs WOOX Pro" 대비를 노출 (브랜드 컬러 강조)

**WOOX Pro 비교 카드 포함 콘텐츠:**

| 요소 | 상세 내용 |
|------|-----------|
| **손실 금액 인지** | 동일 조건에서 WOOX Pro와 현재 거래소의 **토탈 세이빙 기준** 실질 절감액 차이(USDT) 및 차이(%p)를 대비 표시. 예시(단순화, 실제 값은 §2-A-1 "Monthly Estimated Cashback" 엔진 산출): Zoomex 조건에서 현재 대비 WOOX Pro 기준 **월 약 +567 USDT 더 절약**(구체 %p는 엔진 산출값 사용) |
| **표시 방식** | **토탈 세이빙 기준으로 확정**(OI-08, 2026-07-03). "OOO USDT 더 아낄 수 있었습니다/있습니다" 프레이밍 사용 |
| **CTA 버튼 (Primary)** | 비교 카드 내 "WOOX Pro로 시작" 버튼 → WOOX Pro 거래소 상세 페이지 이동 (테더맥스 브랜드 컬러) |
| **기준 명시** | 카드 하단에 "일반 유저/기본 등급 기준" 소문자 표기 |
| **접기** | 카드 접기(collapse) **제공**(OI-05 확정). 강제 닫기(X)는 없음 |

**케이스 B: 선택한 거래소가 WOOX Pro인 경우**

- WOOX Pro 비교 카드를 **삽입하지 않음**
- 기존 정상 페이백 결과 화면만 노출

> **같은 화면에 기능 3도 함께 노출된다**: 이 캐시백 프리뷰 결과 화면(S2)의 인라인 비교 카드 아래에는 **기능 3(WOOX Pro × 빗썸 트레블룰 연동 배너, 위치 #1)**이 케이스 A/B 분기와 별개로 노출된다. 노출 여부는 기능 2와 동일한 공통 프로모션 활성 게이트(§3-3)를 따른다. 상세는 기능 3 섹션 참조.

## 2-A. 기간 한정 추가분 상세 로직 (a~d)

§2의 기능 1·2 중 **프로모션 기간에만 적용**되는 분기 (a)~(d)의 구현 상세이다.
(a)=기능 1 WOOX Pro 미포함, (b)=기능 1 WOOX Pro 포함, (c)=기능 2 타 거래소, (d)=기능 2 WOOX Pro.

### 2-A-1. 절약액 계산 모델 (공통)

유저 화면 계산에는 **커미션율을 사용하지 않는다.** 유저가 실제로 돌려받는 값은 "본인이 낸 수수료 × 페이백율"이므로, 비교는 **페이백율**(또는 할인 포함 **토탈 세이빙율**)만으로 성립한다. 커미션율(90%)·마진율은 백오피스 내부 정산용이며 유저 계산과 무관하다.

> **기능 1·기능 2는 데이터 소스가 완전히 독립적이며 상호 연관이 없다** (2026-07-03 확인). 기능 1은 유저 UID 테이블의 **실제 발생한 과거 데이터**를 역산한다. 기능 2(캐시백 프리뷰)는 UID 테이블과 무관하게, **유저의 실제 거래량을 알 수 없으므로 인풋(자산·레버리지·빈도 등)만으로 미리 정해진 수식이 예측값을 계산**하는 순수 프로젝션이다. 아래 "명목 기준"은 기능 1 전용, "프로젝션 기준"은 기능 2 전용이며 서로 데이터를 공유하지 않는다.

**핵심 지표 정의 (명목 기준 — 기능 1 · 실측 데이터에 적용)**

```
토탈 세이빙율(E) = 1 − (1 − 할인율(E)) × (1 − 페이백율(E))
                = 할인율(E) + (1 − 할인율(E)) × 페이백율(E)   ← 동일 식
```

- WOOX Pro는 할인율 0% → **명목 토탈 세이빙율 = 페이백율 = 80%**
- 백오피스 "Total Saving %" 자동계산 필드와 동일 산식 (메이커/테이커 각각 산출)
- 이 명목값은 **기능 1(실제 발생한 과거 페이백 데이터의 역산)에 사용**한다. 기능 2(미래 예측치인 프리뷰)는 아래 "기능 2 전용 프로젝션 공식"의 보정계수 반영 버전을 사용한다 (2026-07-03 확인, 엔지니어링 산식 반영).

**표시 %p 정의 (기능 1·기능 2 공통 확정, 2026-07-03)**

```
표시 %p = 명목 토탈세이빙율(WOOX Pro) − 명목 토탈세이빙율(현재 거래소)
         (보정계수 0.7은 미반영 — %p는 항상 명목 토탈세이빙율 기준)
```

- USDT 절약 금액은 기능별로 다른 산식(기능 1=실측 역산, 기능 2=보정계수 반영 프로젝션)을 쓰지만, **%p는 두 기능 모두 위 명목 토탈세이빙율 차이로 통일**한다.
- 페이백율 차이와 동일해 보이지만 **할인율이 있는 거래소가 비교 대상이면 값이 달라지므로**, 정확한 정의는 "토탈세이빙율 차이"다.
- 역효과 판정 기준(§3-1)과 동일한 지표를 사용하므로 판정 기준과 표시 %p가 항상 일치한다.

**기준 수수료 역산 (기능 1 · 출금 화면)**

출금 화면에는 유저의 실제 페이백 금액만 존재한다. 거래소 원 수수료율(0.04% 등)은 불필요하며 아래처럼 역산한다.

```
기준 수수료 = 실제 페이백 금액 ÷ ((1 − 할인율) × 페이백율)
             (할인율 0 가정 시:  실제 페이백 금액 ÷ 페이백율)
```

**절약액 표시 방식 — 토탈 세이빙 기준 확정 (OI-08, 2026-07-03)**

C레벨 데모 결과 **토탈 세이빙 기준**으로 최종 확정되었다. 페이백률 기준(대안으로 검토했던 방식)은 폐기한다. 기능 1·기능 2 공통 적용.

```
현재 순비용 = 기준 수수료 × (1 − 현재 토탈세이빙율)
WOOX Pro 순비용 = 기준 수수료 × (1 − WOOX Pro 토탈세이빙율)
추가 절약 = 현재 순비용 − WOOX Pro 순비용
```

문구 프레이밍: "WOOX Pro였다면 수수료를 **OOO USDT** 더 아낄 수 있었습니다/있습니다"

**검증 예시** (비트겟 페이백율 54%, 실제 페이백 54 USDT, 할인율 0% 가정)

- 기준 수수료 = 54 ÷ 54% = **100 USDT**
- 현재 순비용 = 100 × 46% = 46, WOOX Pro 순비용 = 100 × 20% = 20
- **추가 절약 = 26 USDT** → "WOOX Pro였다면 수수료를 26 USDT 더 아낄 수 있었습니다"
- 참고: 할인율이 있는 거래소가 섞이면 페이백률 기준과 결과값이 달라지는데, 토탈 세이빙 기준이 실질 절감액을 정확히 반영하므로 이 방식으로 확정했다.

**기능 2 전용 — 캐시백 프리뷰 프로젝션 공식 (Monthly Estimated Cashback)**

> 신규 산식이 아니라 **기존 캐시백 프리뷰 엔진의 실제 산식**이다(엔지니어링 확인, 2026-07-03). 본 프로젝트는 이 값을 그대로 사용해 WOOX Pro와 비교만 추가하며, 산식 자체를 새로 만들지 않는다.

```
Monthly Estimated Cashback(E)
= balance × leverage × TIME × 2 × 30
× actualFeePaid(E) × 페이백율(E) × 보정계수(E)

actualFeePaid(E) = (Maker 수수료율 × (1 − Maker 할인율(E)) × Maker 비율)
                  + (Taker 수수료율 × (1 − Taker 할인율(E)) × Taker 비율)
```

| 변수 | 설명 |
|---|---|
| balance · leverage | 유저 입력(자산 규모·레버리지) |
| Maker/Taker 비율 | 유저 입력(슬라이더, 합계 100% → 0~1 정규화) |
| Maker/Taker 수수료율·할인율 | 거래소별 어드민 값(Maker/Taker 별도 관리) |
| 페이백율 | 거래소별 **단일값** (Maker/Taker 구분 없음 — 가중합은 수수료율 단계에서만 발생) |
| **보정계수** | **0.7 고정 상수** (전 거래소·전 상품 동일, 영구 상수로 취급 — 값이 바뀌면 전 거래소 동시에 함께 변경됨. 거래소별 개별 조회 불필요, 전역 설정값 하나로 하드코딩) |
| TIME | 거래 빈도 매핑값. **기능 2(캐시백 프리뷰) 전용 개념**이며 전 거래소 공통 적용(거래소별 차이 없음) |

| 빈도 | TIME |
|---|---|
| 0~1회/일 | 0.6 |
| 1~2회/일 | 1.5 |
| 2~5회/일 | 3.0 |
| 5~10회/일 | 7.5 |
| 10회 이상/일 | 20.0 |

**비교 방법**: 동일한 balance·leverage·TIME·Maker/Taker 비율(유저 입력)을 WOOX Pro와 현재 거래소 양쪽에 각각 대입해 거래소별 수수료율·할인율·페이백율(보정계수는 0.7 고정 공통)로 **프리뷰 전용 토탈세이빙율**을 구하고, 그 차이로 절약액을 산출한다 (토탈 세이빙 기준 확정, OI-08):

```
토탈 세이빙율_프리뷰(E) = 1 − (1 − 할인율(E)) × (1 − 페이백율(E) × 0.7)
```

- 보정계수가 전 거래소 동일(0.7 고정)하므로 **비교 결과(어느 쪽이 더 유리한지)에는 영향을 주지 않는다.** 다만 화면에 표시되는 **절대 금액**은 0.7을 반드시 반영해야 실제 캐시백 프리뷰 엔진 결과와 숫자가 일치한다.
- **선행 조건 (별도 엔지니어링 트랙, 본 프로젝트 범위 아님)**: 어드민 "테더맥스 적용(%)" 필드가 현재 할인율에 페이백율(cashbackRate)까지 이중으로 곱해 저장되는 버그가 있음. 기능 2는 이 필드의 할인율 값을 그대로 읽어 계산하므로, **이 버그가 프로모션 런칭 전 수정 완료되어야** 비교 숫자가 정확하다 (⚠️ OI-10).

| WOOX Pro 파라미터 | 값 | 성격 |
|------|------|------|
| 할인율 | 0% | 내부/백오피스 |
| 페이백율(캐시백) | 80% | 내부 전용 (비노출) |
| 명목 토탈 세이빙율 | 80% (할인율 0이므로 페이백율과 동일, 기능 1용) | 내부 전용 (비노출) |
| 보정계수 | **0.7** (전 거래소 공통 고정 상수) | 내부 전용 (비노출) |

- **데이터 소스**: 실제 페이백 금액(기능 1)은 **유저 UID 테이블 값**(하루 1회 배치 업데이트, KST 20:00). 출금은 배치와 무관하게 상시 가능하며, 기능 1은 **출금 시점의 UID 페이백 값**을 역산 기준으로 사용. **기능 2(캐시백 프리뷰)는 이 UID 테이블과 무관** — 유저 인풋만으로 산출하는 순수 프로젝션이다. 거래소별 할인율/페이백율은 기존 캐시백 엔진·백오피스. 보정계수(0.7)는 고정 상수. 모두 내부 보유, 외부 의존 없음.
- **노출 원칙**: 내부 수치(할인율·페이백율·보정계수·커미션율·마진율)는 유저 화면에 절대 표시하지 않고 **결과값(절약 금액 USDT, %p)만** 노출한다. (CLAUDE.md 핵심 비즈니스 규칙 6 참조)
- **시간 표시 원칙**: 배치 시각(KST 20:00)은 서버 내부 처리 기준이다. 배치 시차 관련 안내 문구(NFR-005)를 화면에 노출할 경우, **KST 고정 문구가 아니라 사용자 기기의 로컬 타임으로 변환해 표시**한다 (CLAUDE.md 핵심 비즈니스 규칙 9 참조).

### 2-A-2. (a) WOOX Pro 미포함 출금 → 가상 피드백

| 우선순위 | 노출 콘텐츠 | 조건 |
|------|------|------|
| 1 | 가상 피드백 ("WOOX Pro였다면 +X USDT") | 역효과 미해당(WOOX Pro 토탈세이빙 > 타, §3-1) & 절약액 > 0 & 계산 가능 (양수인데 1 USDT 미만이면 1 USDT로 표시) |

**복수 거래소(UID) 동시 출금**: 거래소별 추가 이득을 각각 계산 → **역효과 미해당 & 양수인 UID만 합산** → **단일 총액 하나로 표시**. 역효과·음수 UID는 합산 제외. 하단에 "N개 거래소 기준·일반 등급 기준" 명시.

- (선택) "내역 보기"로 UID별 절약액을 펼쳐 볼 수 있게 하는 것은 디자인 옵션으로 검토.

**표시 문구**: 계산값은 §2-A-1에서 확정된 **토탈 세이빙 기준**으로 노출한다(OI-08 확정, 2026-07-03).

### 2-A-3. (b) WOOX Pro 포함 출금 → 이벤트 분기

| 우선순위 | 노출 콘텐츠 | 조건 |

**노출 조건 (AND)**: 타 거래소 선택 · 유저 WOOX Pro 미연동 · 정확 비교 가능 · **역효과 미해당(WOOX Pro 토탈세이빙율 > 타 거래소, §3-1)**

- 노출 형태: 결과 카드 **바로 아래 인라인 비교 카드**(팝업·딤 없음). 기능 2 정의 참조.

- WOOX Pro 미지원 조건(레버리지/상품): 미노출 (부정확 비교 차단).

WOOX Pro 비교 카드를 삽입하지 않고 정상 페이백 결과 화면만 노출. (c) 노출 조건 미충족 케이스 전부 포함.

## 3. 예외 처리 및 데이터 정책 (Exception & Data Policy)

### 3-1. 수수료 비교 계산의 정확성 및 신뢰성 확보 (역효과 방지)

- **역효과 판정 기준 = 토탈 세이빙율.** 표시 수치도 토탈 세이빙 기준으로 확정(OI-08)되었으므로, **노출 여부 판정과 표시 수치가 동일한 토탈 세이빙율 산식으로 통일**된다.
- 판정 규칙: **타 거래소의 토탈 세이빙율이 WOOX Pro와 같거나 높은 경우**(타 ≥ WOOX Pro) → **출금 후 광고·프리뷰 비교 카드를 노출하지 않음.** 즉 **WOOX Pro 토탈 세이빙율이 타 거래소보다 엄격히 클 때만(WOOX Pro > 타)** 노출.
  - 예: 타 거래소 최고 VIP 등급 등으로 토탈 세이빙이 WOOX Pro 이상이 되는 케이스는 전부 미노출.
- 비교 기준은 **'일반 유저/기본 등급'** 혹은 명확한 산정 기준을 비교 카드 하단에 아주 작은 텍스트로 명시하여 신뢰도 확보.

### 3-2. 유저 피로도 방어 로직 (UX 방어 로직)

**결정 (OI-04 반영):** 팝업 모달을 폐기하고 **결과 페이지 인라인 변형(결과 카드 아래 WOOX Pro 비교 카드)**으로 전환한다. 인라인은 화면을 가리지 않는 비강제 노출이므로, 조건을 여러 번 바꿔 계산해도 팝업이 반복되며 발생하던 이탈 문제 자체가 사라진다.

| 항목 | 내용 |
|------|------|
| **노출 빈도** | 조건 충족 시 결과 페이지에 **상시 인라인 노출** (팝업 기반 '세션당 1회/하루 보지 않기'는 불필요) |
| **접기 옵션** | 비교 카드 접기(collapse) **제공**(OI-05 확정). 강제 닫기(X) UI 없음 |
| **역효과·미지원 필터** | 타 거래소 토탈세이빙율 ≥ WOOX Pro이거나 정확 비교 불가 시 비교 카드를 **삽입하지 않음** (3-1 참조) |

### 3-3. 프로모션 조기 종료 및 어드민 제어

프로모션은 자동 만료(D+30) 외에도 어드민에서 조기 종료할 수 있다. **조기 종료는 이 프로젝트에서 신규 개발하지 않고, 기존 2개 WOOX Pro 이벤트의 어드민 조회 화면에 있는 종료 버튼으로 처리한다.**

| 항목 | 내용 |
|------|------|
| **자동 종료** | WOOX Pro 거래소 온보딩 시점 기준 D+30 도달 시 자동으로 평상시 로직 롤백 |
| **조기 종료 수단** | 2개 WOOX Pro 이벤트 **각각의 어드민 종료 버튼** (기존 어드민 재사용, 신규 개발 아님) |
| **OR 로직** | **2개 종료 버튼 중 1개만 눌러도 종료 적용** |
| **종료 범위** | 종료 시 **프로모션 3개 기능(기능 1·2·3) 전부 일괄 롤백.** 부분(기능별) 종료 없음 |
| **종료 트리거** | (a) D+30 자동 만료 또는 (b) 2개 이벤트 중 어느 하나의 종료 버튼 클릭 — **먼저 도달하는 시점** |

**종료(또는 D+30) 시 롤백 거동 — 3개 기능 동시:**

| 기능 | 롤백 후 거동 |
|------|------|
| **기능 1** | 출금 완료 후 광고를 기존 "진행 중 거래소 이벤트 중 임의 1개 노출" 로직으로 복귀. 가상 피드백 미노출. |
| **기능 2** | 캐시백 프리뷰 결과에서 WOOX Pro 비교 카드 미삽입. 기존 정상 결과 화면만 노출. |
| **기능 3** | 트레블룰 신뢰 배너를 5개 위치 전부에서 제거. |

**구현 요구사항:**

- **프로모션 활성 판정 = (D+30 이내) AND (2개 WOOX Pro 이벤트 모두 미종료).** 어느 하나라도 종료되거나 D+30이 지나면 **3개 기능 즉시 base 로직 복귀**.
- 개발·배포는 기능별 병렬 가능하나, **런타임 종료는 위 단일 게이트로 3기능 일괄** (기능별 개별 종료 없음).
- 종료 시점에 이미 렌더링된 화면의 처리 방식, 어드민 종료 권한 범위, 종료 후 재개(재활성화) 가능 여부는 추후 정의한다.
