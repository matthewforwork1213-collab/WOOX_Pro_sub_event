## 기능 상세

### F-001. 출금 완료 후 가상 피드백 계산 (WOOX Pro 미포함)

| 항목 | 내용 |
|---|---|
| 설명 | 출금 UID 목록을 입력받아, 실제 페이백 금액을 역산해 WOOX Pro 가상 피드백 절약액을 산출한다 |
| 입력 | 출금 UID 목록(복수 가능), 유저 식별자 |
| 처리 | 1) F-005로 프로모션 활성 여부 선확인, 비활성이면 즉시 `노출여부=false` 반환 2) UID별로 유저 UID 테이블에서 실제 페이백 금액 조회(일 1회 KST 20:00 배치 데이터) 3) UID별 거래소의 할인율·페이백율(백오피스)로 `기준 수수료 = 실제 페이백 ÷ ((1−할인율)×페이백율)` 역산 4) UID별 **토탈 세이빙 기준**(OI-08 확정) 순비용 차이 계산 — 산식은 PRD §2-A-1 5) UID별 타 거래소 토탈세이빙율 ≥ WOOX Pro 명목 토탈세이빙율(80%)이면 해당 UID를 역효과로 마킹, 절약액 합산에서 제외 6) 역효과 아닌 UID 중 절약액이 양수인 것만 합산 7) 합산액이 1 USDT 미만이면 1 USDT로 보정, 0 이하면 `노출여부=false` |
| 출력 | `노출여부`(bool), `절약액`(USDT, 토탈 세이빙 기준), `절약%p`(= 명목 토탈세이빙율(WOOX Pro) − 명목 토탈세이빙율(현재 거래소), 보정계수 미반영. PRD §2-A-1), `거래소수`(N, 합산에 포함된 UID 수) |
| 예외 처리 | 배치 미반영 등으로 특정 UID의 페이백 데이터가 없으면 해당 UID는 계산에서 제외(에러로 처리하지 않음). 전체 UID가 계산 불가하면 `노출여부=false`. 배치 시차 안내 문구를 화면에 노출할 경우, 배치 기준 시각(KST 20:00)은 서버 내부 값이며 **API는 UTC 타임스탬프로 응답**하고 FE가 사용자 기기의 로컬 타임으로 변환해 렌더링한다(NFR-007) |
| 관련 화면 | S1 (출금 완료 화면) |
| 관련 API | GET /api/promo/withdrawal-feedback (잠정) |

### F-002. 출금 완료 후 WOOX Pro 이벤트 분기 판정

| 항목 | 내용 |
|---|---|
| 설명 | WOOX Pro가 포함된 출금 건에 대해 노출할 이벤트 유형을 우선순위에 따라 판정한다 |
| 입력 | 출금 UID 목록(WOOX Pro 포함 여부 판별용), 어드민에 등록된 이벤트 목록 |
| 처리 | 1) F-005로 프로모션 활성 여부 확인 2) WOOX Pro 자체 이벤트가 어드민에 활성 등록되어 있으면 `이벤트유형=woox_event` 3) 없으면 테더맥스 WOOX Pro 온보딩 이벤트 활성 여부 확인 → 있으면 `이벤트유형=onboarding_event` 4) 둘 다 없으면 `이벤트유형=house_ad` (경쟁 거래소 이벤트는 후보에서 제외) |
| 출력 | `이벤트유형`(`woox_event` / `onboarding_event` / `house_ad`), 해당 이벤트 메타데이터(배너 이미지·문구 등은 기존 이벤트 어드민 데이터 재사용) |
| 예외 처리 | 프로모션 비활성 시 F-001과 동일하게 base 로직(`house_ad`와 동일 취급)으로 반환 |
| 관련 화면 | S1 (출금 완료 화면) |
| 관련 API | GET /api/promo/withdrawal-feedback (F-001과 동일 엔드포인트, WOOX Pro 포함 여부에 따라 응답 케이스 분기) |

### F-003. 캐시백 프리뷰 WOOX Pro 비교 계산

| 항목 | 내용 |
|---|---|
| 설명 | 유저가 입력한 프리뷰 조건으로 현재 거래소와 WOOX Pro 각각의 Monthly Estimated Cashback을 계산해 비교 결과를 반환한다 |
| 입력 | 거래소(현재 선택), balance(자산 규모), leverage, Maker/Taker 비율, 거래 빈도(TIME 매핑용) |
| 처리 | 1) F-005로 프로모션 활성 여부 선확인 2) 입력값과 현재 거래소의 Maker/Taker 수수료율·할인율·페이백율로 `actualFeePaid`, `Monthly Estimated Cashback`(보정계수 0.7 고정) 계산 — 기존 캐시백 프리뷰 엔진 산식 재사용 3) 동일 입력값에 WOOX Pro 파라미터(할인율 0%, 페이백율 80%, 보정계수 0.7)를 대입해 WOOX Pro 값 계산 4) 프리뷰 전용 토탈세이빙율 `1−(1−할인율)×(1−페이백율×0.7)` 기준으로 역효과 판정(타 거래소 ≥ WOOX Pro면 역효과) 5) 레버리지·상품이 WOOX Pro 미지원 조건이면 별도로 미노출 처리 6) 역효과·미지원이 아니면 **토탈 세이빙 기준**(OI-08 확정) 순비용 차액을 절약 금액(USDT)으로 산출 7) `절약%p`는 별도로 **명목 토탈세이빙율**(보정계수 미반영, `1−(1−할인율)×(1−페이백율)`) 차이로 계산 — USDT 금액 계산과 다른 산식 사용 |
| 출력 | `노출여부`(bool), `현재거래소_예상값`, `WOOXPro_예상값`, `절약액`(USDT, 토탈 세이빙 기준·보정계수 반영), `절약%p`(= 명목 토탈세이빙율(WOOX Pro) − 명목 토탈세이빙율(현재 거래소), 보정계수 미반영. 기능 1과 동일 정의, PRD §2-A-1) |
| 예외 처리 | 거래소=WOOX Pro가 이미 선택된 요청은 애초에 이 API를 호출하지 않는 것을 FE 계약으로 하되, 서버에서도 방어적으로 `노출여부=false` 반환. 캐시백 엔진의 "테더맥스 적용" 필드 이중곱 버그(OI-10)가 미수정 상태면 계산 결과 정확도가 보장되지 않으므로, 버그 수정 완료 전 배포하지 않는다(런칭 전 선행 조건) |
| 관련 화면 | S2 (캐시백 프리뷰 결과 페이지) |
| 관련 API | POST /api/promo/cashback-preview/compare (잠정) |

### F-004. 트레블룰 배너 노출 판정

| 항목 | 내용 |
|---|---|
| 설명 | 5개 위치의 트레블룰 배너 노출 여부와 노출 문구(i18n 키)를 결정한다 |
| 입력 | 화면 진입 요청(위치 무관, 공통 판정), 유저 로케일 |
| 처리 | 1) F-005로 프로모션 활성 여부 조회 2) 활성이면 `노출여부=true`와 함께 i18n 키(예: `promo.travelrule.badge`)를 반환 3) 로그인 전 요청(S3·S4)도 유저 식별자 없이 동일 판정 로직 적용(세션 무관) |
| 출력 | `노출여부`(bool), i18n 키 |
| 예외 처리 | 프로모션 비활성 시 `노출여부=false`. 로고 이미지 자체는 API로 내려주지 않음(디자인 하드코딩, REQ-017) |
| 관련 화면 | S1~S5 (App은 S1·S2만 해당) |
| 관련 API | GET /api/promo/status (F-005와 동일 엔드포인트) |

### F-005. 프로모션 활성/종료 판정 및 일괄 롤백

| 항목 | 내용 |
|---|---|
| 설명 | 프로모션 전체의 활성/비활성 상태를 단일 기준으로 판정하고, F-001~F-004가 공통으로 참조하는 게이트를 제공한다 |
| 입력 | 없음(서버 내부 상태 조회) — WOOX Pro 온보딩 기준일, 2개 WOOX Pro 이벤트의 어드민 종료 상태 |
| 처리 | `활성 = (현재시각 ≤ WOOX Pro 온보딩 기준일 + 30일) AND (2개 WOOX Pro 이벤트가 모두 종료되지 않음)`. 2개 이벤트 중 하나라도 어드민에서 종료되면 즉시 `활성=false`로 전환(OR 로직, 신규 종료 UI 개발 없이 기존 이벤트 종료 상태를 그대로 조회) |
| 출력 | `활성`(bool) — F-001~F-004가 공통으로 소비하는 단일 값 |
| 예외 처리 | 이벤트 종료 상태 조회 실패 등 판정 불가 상황에서는 안전하게 `활성=false`로 처리(장애 시 프로모션 넛지가 노출되지 않는 방향으로 폴백) |
| 관련 화면 | S1~S5 전체 (공통 게이트) |
| 관련 API | GET /api/promo/status (잠정) |
