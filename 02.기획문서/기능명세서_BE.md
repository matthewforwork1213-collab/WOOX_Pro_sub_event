## 0. 문서 사용 안내 (FE/BE 구분 정의)

- **FE/BE는 "개발 영역" 구분이다** (프론트엔드 코드 / 백엔드 코드). 유저 프론트 vs 백오피스(어드민) 구분이 아니다.
- 본 프로젝트의 5개 기능(F-001~F-005)은 **전부 유저 프론트 기능**이며, **동일한 기능을 화면 담당(FE)과 계산·데이터 담당(BE)이 나눠** 명세한다. 하나의 기능이 FE·BE 양쪽 문서에 모두 등장한다(예: F-003 캐시백 비교 = FE는 카드 삽입/접기, BE는 금액 계산).
- **백오피스(어드민)는 본 프로젝트에서 신규 개발이 없다.** 기존 어드민 값(할인율·페이백율·이벤트 등록/종료 상태)을 **읽기만** 한다. 어드민 의존성 요약은 §1.6·§6 참조.

## 1. 공통 정책 (F-001~F-005 공통, BE)

모든 기능은 아래 공통 규칙을 전제로 한다. 각 기능 상세(§3)는 이 절을 반복 서술하지 않고 참조한다.

### 1.1 노출 제어 — 백오피스 영역별 on/off (2026-07-07 정책)

> **정책 변경(2026-07-07)**: 기존 "D+30 AND 2개 WOOX Pro 이벤트 미종료" 공통 게이트를 **폐지**하고, 각 노출 영역을 **백오피스 on/off로 독립 제어**한다. 프로모션 기간도 어드민(백오피스)에서 통제한다 — C레벨 결정으로 OI-07 갱신(기존 30일 한정·상시화 미적용 대체, 2026-07-07 확정).

- 각 영역은 대응하는 백오피스 플래그가 **ON일 때만** 노출한다(F-005가 5개 플래그를 제공). 부분 제어 허용(영역별 독립).

| 영역 | 백오피스 플래그 | OFF일 때 동작 |
|---|---|---|
| 출금완료 WOOX Pro 가상 피드백 (F-001) | `s1Feedback` | 가상 피드백 미표시 → 원 base 이벤트 로직(진행 중 거래소 이벤트 임의 1개) |
| 출금완료 트레블룰 배너 #2 (F-004) | `s1Banner` | 배너 미노출 |
| 캐시백 프리뷰 WOOX Pro 비교 (F-003) | `s2Compare` | 비교 카드 미삽입 |
| 캐시백 프리뷰 트레블룰 배너 #1 (F-004) | `s2Banner` | 배너 미노출 |
| 로그인 3페이지 트레블룰 #3·#4·#5 (F-004) | `loginBanners` | 세 위치 일괄 미노출 |

- 각 플래그는 **독립적**이다(특정 영역만 OFF 가능). 기존 "3기능 일괄 롤백·부분 종료 없음"은 **폐지**(REQ-021 개정).
- 신규 종료 UI 없이 **신규 백오피스 노출 제어 페이지**의 on/off로 관리한다(기존 2개 WOOX Pro 이벤트 종료 버튼 종속 폐지, REQ-022).
- WOOX Pro 포함 출금의 이벤트 분기(F-002)는 `s1Feedback`과 무관하게 동작(이벤트 분기는 가상 피드백 안내와 별개).

### 1.2 시간 처리 (KST 내부 / UTC 응답)

- 배치 처리(KST 20:00)는 **서버 내부적으로 KST 절대시각** 기준으로 계산한다. (2026-07-07 D+30 자동 판정 폐지 — 기간은 백오피스 통제)
- **API 응답의 모든 타임스탬프 필드는 UTC(ISO 8601)로 내려준다.** 화면 표시용 로컬 타임 변환은 FE가 수행한다. 서버는 KST 고정 문구를 응답에 담지 않는다(NFR-007).

### 1.3 내부 수치 비노출 (응답 페이로드 규칙)

- 할인율·페이백율·보정계수·커미션율·마진율 등 **내부 원자료는 어떤 API 응답에도 포함하지 않는다**(REQ-023, NFR-002).
- 유저 대상 응답에는 **계산 결과값만**(절약 금액 USDT, %p, 예상 캐시백 등) 담는다.

### 1.4 역효과 방지 판정 (F-001·F-003 공통)

- 비교/피드백 노출은 **WOOX Pro가 엄격히 유리할 때만** 한다.
- 판정 지표는 **토탈세이빙율**이며, 타 거래소 토탈세이빙율이 **WOOX Pro와 같거나 높으면(≥) 미노출**(`visible=false`). WOOX Pro가 **초과(>)**일 때만 노출한다.
- 기능 1은 **명목** 토탈세이빙율, 기능 2는 **프리뷰 전용** 토탈세이빙율(보정계수 반영)로 판정한다(§2.2). 단 표시 %p는 두 기능 모두 명목 기준(§2.3).

### 1.5 노출 게이팅과 표시 하한 (혼동 주의)

- **게이팅 기준은 "절약액 > 0"**(양수면 노출 후보). 1 USDT는 게이팅 임계값이 **아니다.**
- **표시 하한은 1 USDT**: 절약액이 양수인데 1 USDT 미만이면 **서버가 1 USDT로 보정**해 반환한다. 0 이하이면 `visible=false`(REQ-003).

### 1.6 데이터 소스 (기능별 독립)

| 기능 | 데이터 소스 | 갱신 |
|---|---|---|
| F-001 | 유저 UID 테이블의 **실제 발생 페이백 금액**(과거 실측) | 일 1회 배치, **KST 20:00** |
| F-003 | **유저 입력값만**(UID 테이블과 무관한 순수 프로젝션) | 요청 시 실시간 계산 |
| 공통(거래소 파라미터) | 어드민/캐시백 엔진의 거래소별 Maker/Taker 수수료율·할인율·페이백율 | 어드민 관리 |

- **F-001과 F-003은 데이터 소스가 완전히 독립적이다.** F-003은 유저의 실제 거래량을 알 수 없으므로 입력값 기반 예측만 수행한다(UID 테이블 조회 없음).
- 출금은 배치와 무관하게 상시 가능하며, F-001은 **출금 시점의 UID 페이백 값**을 역산 기준으로 쓴다.

## 2. 계산 모델 및 상수 (BE 핵심)

### 2.1 상수 · WOOX Pro 파라미터

| 항목 | 값 | 성격 |
|---|---|---|
| WOOX Pro 할인율 | 0% | 내부/백오피스 |
| WOOX Pro 페이백율(캐시백) | 80% | 내부 전용(비노출) |
| WOOX Pro 명목 토탈세이빙율 | 80% (할인율 0이라 페이백율과 동일) | 내부 전용(비노출) |
| **보정계수(Correction Factor)** | **0.7 고정 상수** | 내부 전용(비노출) |

- 보정계수는 **전 거래소·전 상품 공통 고정 상수**다(영구 상수 취급). 거래소별 개별 조회가 아니라 **전역 설정값 하나를 하드코딩**한다. 값이 바뀌면 전 거래소가 동시에 함께 바뀐다.
- 보정계수는 **기능 2(프리뷰 예측)에만** 곱한다. **기능 1(실측 역산)에는 적용하지 않는다.**

### 2.2 토탈세이빙율 정의

**명목(기능 1 · 실측 데이터에 적용)**
```
토탈세이빙율(E) = 1 − (1 − 할인율(E)) × (1 − 페이백율(E))
```
- WOOX Pro: 할인율 0 → 명목 토탈세이빙율 = 페이백율 = 80%.

**프리뷰 전용(기능 2 · 예측치에 적용, 보정계수 반영)**
```
토탈세이빙율_프리뷰(E) = 1 − (1 − 할인율(E)) × (1 − 페이백율(E) × 0.7)
```
- 보정계수가 전 거래소 동일(0.7)하므로 **어느 쪽이 유리한지(비교 방향)에는 영향이 없다.** 다만 화면에 표시되는 **절대 금액**은 0.7을 반영해야 실제 캐시백 엔진 결과와 숫자가 일치한다.

### 2.3 표시 %p 정의 (기능 1·2 공통, 보정계수 미반영)

```
표시 %p = 명목 토탈세이빙율(WOOX Pro) − 명목 토탈세이빙율(현재 거래소)
         (보정계수 0.7은 미반영 — %p는 항상 명목 기준)
```
- USDT 절약 금액은 기능별 산식이 다르지만(§2.4/§2.5), **%p는 두 기능 모두 이 명목 산식으로 통일**한다(REQ-024).
- 역효과 판정(§1.4)과 동일한 토탈세이빙율 지표이므로 판정 기준과 표시 %p가 항상 정합한다.
- 페이백율 차이와 같아 보이지만 **할인율이 있는 거래소가 비교 대상이면 값이 달라지므로**, 정확한 정의는 "토탈세이빙율 차이"다.

**복수 UID 합산 시 단일 %p (기준 수수료 가중평균, 2026-07-06 확정)**

기능 1에서 복수 거래소 UID를 합산할 때는 단일 %p를 아래 **기준 수수료 가중평균**으로 산출한다(단순 합산·산술평균 금지). 합산 대상은 `savingAmount`와 동일 집합(역효과 미해당 & 절약액 양수 UID)이다.
```
현재 실효 토탈세이빙율 = Σ(기준수수료_i × 현재_i 명목 토탈세이빙율) ÷ Σ(기준수수료_i)
복수 UID %p = 명목 토탈세이빙율(WOOX Pro, 80%) − 현재 실효 토탈세이빙율

  ⇔ (동치 항등식)  복수 UID %p = 추가 절약 총액 ÷ Σ(기준수수료_i)
```
- WOOX Pro 순비용은 UID마다 `기준수수료 × 0.2`이므로 WOOX Pro 실효 토탈세이빙율은 항상 80%. 따라서 위 두 식은 동일하다.
- 단일 UID(N=1)이면 이 식은 §2.3 기본 정의와 정확히 같아진다.
- %p 계산에는 1 USDT 표시 하한 **보정 전** 실제 합산값(추가 절약 총액·Σ기준수수료)을 사용한다.

### 2.4 기능 1 — 기준 수수료 역산 및 절약액

출금 화면에는 유저의 **실제 페이백 금액**만 존재한다. 거래소 원 수수료율은 불필요하며 아래처럼 역산한다.
```
기준 수수료   = 실제 페이백 금액 ÷ ((1 − 할인율) × 페이백율)     (할인율 0 가정 시: 실제 페이백 ÷ 페이백율)
현재 순비용   = 기준 수수료 × (1 − 현재 토탈세이빙율)
WOOX Pro 순비용 = 기준 수수료 × (1 − WOOX Pro 토탈세이빙율)
추가 절약     = 현재 순비용 − WOOX Pro 순비용
```
**검증 예시** (비트겟 페이백율 54%, 실제 페이백 54 USDT, 할인율 0% 가정)

| 단계 | 계산 | 값 |
|---|---|---|
| 기준 수수료 (역산) | 54 ÷ 0.54 | 100 USDT |
| 현재 순비용 | 100 × (1 − 0.54) | 46 USDT |
| WOOX Pro 순비용 | 100 × (1 − 0.80) | 20 USDT |
| 추가 절약 (`savingAmount`) | 46 − 20 | **26 USDT** |
| `savingPercentPoint` | 명목 TS 차이 = 80% − 54% (§2.3) | **26%p** |

→ "WOOX Pro였다면 수수료를 26 USDT(26%p) 더 아낄 수 있었습니다"

⚠️ **`savingPercentPoint`는 `savingAmount`에 비례하지 않는다.** 위 단일 UID에서 우연히 둘 다 26일 뿐이며, %p는 항상 명목 토탈세이빙율 차이(§2.3)로만 계산하고 금액과 무관하다.

**복수 UID 예시** (Bitget 54% + Zoomex 60%, 각 실제 페이백 54·36 USDT, 할인율 0)

| UID | 실제 페이백 | 기준 수수료 | 현재 순비용 | WOOX 순비용 | 추가 절약 |
|---|---|---|---|---|---|
| Bitget (54%) | 54 | 100 | 46 | 20 | 26 |
| Zoomex (60%) | 36 | 60 | 24 | 12 | 12 |
| **합** | | **160** | | | **38** |

- `savingAmount` = **38 USDT** (표시 하한 보정 후)
- `savingPercentPoint` = 추가절약 총액 ÷ Σ기준수수료 = 38 ÷ 160 = **23.8%p** (§2.3 기준수수료 가중평균 — 산술평균 27%p도, 금액 38도 아님)

### 2.5 기능 2 — Monthly Estimated Cashback (프리뷰 프로젝션)

> 신규 산식이 아니라 **기존 캐시백 프리뷰 엔진의 실제 산식**이다. 본 프로젝트는 이 값을 그대로 사용해 WOOX Pro와 비교만 추가한다.

```
Monthly Estimated Cashback(E)
  = balance × leverage × TIME × 2 × 30
  × actualFeePaid(E) × 페이백율(E) × 보정계수(0.7)

actualFeePaid(E) = (Maker 수수료율 × (1 − Maker 할인율(E)) × Maker 비율)
                 + (Taker 수수료율 × (1 − Taker 할인율(E)) × Taker 비율)
```

| 변수 | 설명 |
|---|---|
| balance · leverage | 유저 입력(자산 규모·레버리지) |
| Maker/Taker 비율 | 유저 입력(합계 100% → 0~1 정규화) |
| Maker/Taker 수수료율·할인율 | 거래소별 어드민 값(Maker/Taker 별도 관리) |
| 페이백율 | 거래소별 **단일값**(Maker/Taker 구분 없음 — 가중합은 수수료율 단계에서만 발생) |
| 보정계수 | **0.7 고정 상수**(§2.1) |
| TIME | 거래 빈도 매핑값(**기능 2 전용**, 전 거래소 공통) |

**TIME 매핑표** (요청의 `dailyTradeFrequency` enum ↔ TIME)

| 빈도 | enum | TIME |
|---|---|---|
| 0~1회/일 | `0_1` | 0.6 |
| 1~2회/일 | `1_2` | 1.5 |
| 2~5회/일 | `2_5` | 3.0 |
| 5~10회/일 | `5_10` | 7.5 |
| 10회 이상/일 | `10_PLUS` | 20.0 |

**비교/절약액 산출**: 동일 입력값(balance·leverage·TIME·Maker/Taker 비율)을 WOOX Pro와 현재 거래소 양쪽에 대입해 각각의 프리뷰 전용 토탈세이빙율(§2.2)로 순비용을 구하고 그 차액을 절약액(USDT, 0.7 반영)으로 산출한다. `절약%p`는 별도로 **명목** 토탈세이빙율 차이(§2.3, 0.7 미반영)로 계산하므로 `savingAmount`와 단순 비례하지 않는다.

- **`savingAmount`의 정답 정의는 "순비용 차액"이다**(위 산식). 비교 거래소의 **할인율이 0인 경우(본 생태계 대부분)** 이 값은 `wooxProEstimate − currentExchangeEstimate`(캐시백 추정 차이)와 정확히 일치한다 — API-003 예시(Zoomex, 할인율 0)의 `567 = 1512 − 945`가 이 경우다. **할인율 > 0인 거래소**가 비교 대상이면 순비용 차액은 할인 부분까지 포함하므로 캐시백 차이와 달라진다. 이때는 **순비용 차액이 정답**이며 `wooxProEstimate − currentExchangeEstimate`를 그대로 쓰지 않는다.

- **선행 조건(별도 엔지니어링 트랙, 본 프로젝트 범위 아님)**: 어드민 "테더맥스 적용(%)" 필드가 할인율에 페이백율(cashbackRate)까지 이중으로 곱해 저장되는 버그가 있다. 기능 2는 이 필드의 할인율을 그대로 읽어 계산하므로 **이 버그가 런칭 전 수정 완료되어야** 숫자가 정확하다(⚠️ OI-10). 미수정 시 이 API를 프로덕션에 배포하지 않는다.
